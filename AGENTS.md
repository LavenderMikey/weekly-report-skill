# AGENTS.md — 周报生成（Codex / 通用 Agent 入口）

> 这是给 **OpenAI Codex CLI** 及其它读取 `AGENTS.md` 的 agent 准备的入口。
> 它和 [SKILL.md](SKILL.md)（Claude Code 入口）共用同一套 `scripts/` 与
> `references/`，只是触发方式不同：Claude Code 会按 SKILL.md 的 description
> **自动唤起**；Codex 没有自动发现机制，请在对话里说一句"按 AGENTS.md 的周报
> 流程帮我写周报"来引导，或直接把历史周报 + 本周材料丢进来。

当用户想**写 / 生成 / 补全一份周报（周报 / weekly report / 工作周报）**，尤其是
提供了过去的周报想模仿其风格、要总结本周工作、或要把本周读的文章/论文整理进
学习板块时，按下面的流程来。

## 核心原则

生成的周报要让领导**看不出是自动写的**，要同时还原三样东西：**视觉格式**（字体、
字号、缩进、行间距、自动编号）、**文字/内容风格**，以及周报的本质——**它是总结性
的语言**。**绝不编造未发生的工作、指标或文章**，宁可朴素也不夸大。

周报是**凝练的总结**，不是把用户发来的东西原样粘贴。无论发来的是论文、长文档、
零散笔记还是代码 diff，都要**先读懂、再凝练**成几句他自己语气的总结。所以有两种
保真度，形式只是其一：

- **形**：字体、行距、编号、排版（好量化的部分）。
- **神**：*他怎么总结*——术语、抽象到多高 / 多技术、句式、怎么落到"启发/结论"、
  压缩到多短。这就是为什么你必须**读懂历史周报的内容，而不仅仅是它的格式**。字体
  对了但语气不对，照样像机器写的。

## 先选路径：能走快路就别走长路

用户最常抱怨"太长了，总在转格式"——通常是因为整条管线（研究 → 克隆 →
LaTeX→Word → PDF→图）跑全了，而其中一半本不需要。先判断走哪条：

- **🟢 快路（常见，重复周）**：用户已有一份目标格式的周报——基本就是**上周的
  `.docx`，尤其是本流程上次生成的那份**。格式已经在那个文件里了，**不要再转储研究
  格式**：直接克隆它、换上本周内容、保存，**跳过第 1 步的格式转储**（`inspect_docx`）。
  但仍要**读一遍上周的正文**抓住语气和总结风格——这只是读文字、很便宜，却是让新一
  周仍然像用户写的关键。这是约八成的情况，应当接近一步到位。
- **🔵 长路（第一次，或没有现成的同格式周报）**：先研究一份历史周报学结构 + 格式
  （第 1 步），再克隆。

**两个重活儿是按需触发，不是默认**：LaTeX→Word 公式（Pandoc）与 PDF→图裁剪
（PyMuPDF）**只在报告确实有公式 / 用户确实想放论文图时才跑**。普通工作周报两个
都不碰——别反射式地调用。光这一点就能去掉大半"总在转格式"的感觉。

## 工作流（5 步）

1. **研究历史周报** → 学结构 + 格式。**（仅长路；重复周跳过）** 已有同格式周报时
   不必转储研究，克隆会继承全部格式；只在**第一次**见到该用户的模板、或克隆结果
   看着不对时才做这步。
   - `.docx`：跑 `python scripts/inspect_docx.py "旧周报.docx"`，它会逐段打印
     样式、字体、字号、对齐、缩进、行间距/段距、以及隐藏的自动编号
     （`num(id=,lvl=)`）。这是你的"格式指纹"。多份周报时，以**最近一份**为
     格式权威模板。
   - `.pdf` / 纯文本：直接读，从可见结构推断格式。
   - **不只看形，更要读懂内容**：认真读历史周报的正文，学它**怎么总结**——提到一篇
     论文/一项工作时用几句话、抽象到什么程度、用什么术语、怎么落到"启发/结论"。
     这套总结风格就是第 3 步要复刻的，比字体更决定"像不像他写的"。
2. **向用户要本周材料**：① 本周工作（文件/文档/要点都行）；② 看过的文章/论文
   （链接、PDF、笔记）。材料不足以填满模板要求的板块时，**追问**，不要拿编造的
   内容凑数。
   - **找出空缺并主动提议预测**：把模板的板块和用户实际给的材料对一遍。有些板块
     是**前瞻/推断性**的——下周计划、心得思考、存在的问题、下一步打算——用户常常
     没写。既不要直接留空（把活又丢回给用户），也不要偷偷编造，而是**明确询问
     用户：要不要根据本周工作 + 历史周报的规律，帮他预测/草拟这些部分**（让用户
     选：预测 / 留空 / 我自己说）。同意就在第 3 步给出有依据的草稿并标注待确认；
     不要就留 `【待补充：下周计划】` 这样的占位标记。
3. **读懂再用他的语气总结**（这是核心）：周报是总结性语言，对每一份输入（论文/
   长文档/笔记/diff）做法都一样——**先读懂，再压缩成几句符合第 1 步语气与详略的
   总结**。绝不原样粘贴源文或摘要，也不要只罗列读了什么而不凝练；一两句话说不清
   它"是什么意思"，说明还没读懂，回去再读。工作板块按模板的分组方式（按项目/主题/
   平铺）组织，突出成果但只写材料支持的内容；学习板块**单独成节**，每篇论文先读懂，
   再用他惯用的术语和技术深度写 1–3 行：是什么、核心思想、启发/与工作的关联，别抄
   摘要，要"用他会说的话"说。
   - **前瞻/预测性板块**（下周计划、心得思考、下一步打算等）：仅在用户第 2 步
     同意时才草拟，否则留占位。草拟时**每条预测都要有依据**——本周在办事项与自然
     的后续，加上这些板块在历史周报里惯常的样子（用户通常写什么、详略与措辞）；
     写完明确标为草稿，请用户确认或改写。这是**唯一**允许推断的地方（前瞻 + 用户
     主动选择），但**不放松下面这条铁律**。
4. **同格式产出**（`.docx` 强烈建议**克隆模板**而非从零重建，见下）。
5. **交付**：给出文件路径 + 各板块填了什么 + 需用户核对处（尤其下周计划和材料
   单薄处）。

## .docx 的关键做法：克隆模板，别从零搭

真实周报带着很多文本看不见的格式：自动编号（存在 `numbering.xml`）、`Normal`
样式上的行距/段距/对齐、页边距、字体。从零重建很容易悄悄丢掉编号或行距。克隆
则全部免费继承。

**克隆—重建配方**（用 `scripts/docx_report.py` 的积木）：

1. `shutil.copy` 最近一份周报到输出路径，再用 `python-docx` 打开。
2. 从最近一周里**各采一个"原型"段落**（深拷贝其 `<w:p>`）：一级编号标题、
   小节编号标题、正文段、计划项、标题行。每个原型自带 `pPr`（含驱动编号的
   `numPr`）和 run 的 `rPr`。
3. `wipe_body(doc)` 清空正文里的所有块级元素（段落和**上几周残留的表格**），只留
   末尾的 `<w:sectPr>`；新内容用 `emit(doc, sectPr, el)` 插在它前面。
4. 用 `clone_with_text(原型, 文本)` / `clone_with_omath(原型, 公式)` 逐行重建。
   复用同一个 `numId` 的标题会被 Word 自动重新编号（综述 + 每篇论文 → 一、二、
   三；每篇论文一个小节 numId → 1、2、3；计划项用计划 numId）。正文/公式行不带
   编号（克隆正文原型，通常 `numId=0`）。
5. `doc.save(...)` 后调用 `strip_unused_media(path)` 删掉 `word/media/` 里没被
   引用的旧图，否则文件会胀到模板原大小（常 10MB+）。

**公式（仅当报告确实有公式）**：普通工作周报跳过，别为了没有的公式去跑 Pandoc。
有公式时渲染成**可编辑的原生 Word 公式**，不要留 LaTeX 源码。用
`scripts/docx_math.py` 的 `latex_to_omath(...)`（经 Pandoc 转 OMML），再
`paragraph._p.append(element)`。需要 PATH 上有 Pandoc；没有就退回保留 LaTeX 文本
并告知用户。

**论文配图（仅当报告确实要放论文图）**：很多周根本不需要图；不确定就用一句
是/否问一下，别擅自抠图。需要时、且源材料是 PDF，用 `scripts/extract_pdf_figure.py` 的
`extract_figure(pdf, page, caption, out_png)` 按图注裁剪。**每张生成的 PNG 都要
打开核对裁剪**。然后用 `docx_report.add_figure_table(...)`（带框的居中表格：图 +
图注）或 `add_centered_image(...)`（无框居中图）插入，看历史周报用哪种就用哪种。
**按模板的位置放图**：架构图放"整体框架/方法"小节下，结果图放"实验结果"下，按
出现顺序编号（图1、图2…），别一股脑堆在末尾。拿不到 PDF 就放 `【此处插入图：…】`
占位并请用户贴截图。

中文字体坑：`python-docx` 里 `run.font.name` 只设拉丁字体，中文要另设 run 的
`w:eastAsia`。详见 [references/docx_formatting.md](references/docx_formatting.md)。
